<objective>
Rewrite the execute workflows to read plans from GitHub, update task sub-issues per-task during execution, post summaries as GitHub comments, and manage board transitions through the full execution lifecycle.
</objective>

<execution_context>
The execute workflow has a thin orchestrator (execute.md, 422 lines) and a per-plan execution sub-workflow (execute-plan.md, 963 lines). There's also execute-phase.md (716 lines) as the main execution orchestrator. Currently these read PLAN.md files from local disk, write SUMMARY.md locally, and don't interact with GitHub.

Locked decisions:
- WIRE-04: Per-task GitHub updates during execution (In Progress when started, Done when completed)
- WIRE-02: No local SUMMARY.md files -- post as GitHub comments
- WIRE-08/09: Issue lifecycle state machine + task sub-issue board tracking
- WIRE-07: On review failure, reopen task sub-issue, move back to In Progress

These are large markdown files. Changes replace local file read/write instructions with MCP tool call instructions.
</execution_context>

<task id="02" type="auto">
  <files>
    - templates/workflows/execute-plan.md

  <action>
Read the full file first (963 lines).

**1. Plan loading -- Read from GitHub instead of local file:**

Replace instructions to read `$phase_dir/{phase}-{NN}-PLAN.md` with:
- Read plan content from the GitHub Issue comment (passed by execute.md orchestrator or fetched via `mcp_get_issue` comments)
- Parse the plan's frontmatter, tasks, and verification criteria from the comment body
- The plan format is identical -- it's just stored as a GitHub comment instead of a file

**2. Per-task board transitions (WIRE-04, WIRE-09):**

Add MCP tool calls at task lifecycle boundaries:

a) **When a task starts:**
  - Look up the task's sub-issue number from init context `task_mappings` or the mapping cache
  - Call `mcp_move_issue` to move the task sub-issue to "In Progress"

b) **When a task completes successfully:**
  - Call `mcp_post_completion` on the task sub-issue (commit SHA, files changed)
  - Call `mcp_close_issue` on the task sub-issue
  - Call `mcp_move_issue` to move to "Done"

c) **When a task fails review:**
  - Call `mcp_reopen_issue` on the task sub-issue
  - Call `mcp_move_issue` to move back to "In Progress"
  - Post a review failure comment via `mcp_post_comment`

**3. Summary posting -- Post to GitHub instead of local file:**

Replace instructions to write `$phase_dir/{phase}-{NN}-SUMMARY.md` with:
- Build summary content in memory (same format as before)
- Call `mcp_post_comment` with `type: 'summary'` on the phase issue
- Include requirement evidence, review cycle data, metrics

**4. Verification posting -- Post to GitHub instead of local file:**

Replace instructions to write VERIFICATION.md or UAT.md with:
- Call `mcp_post_comment` with `type: 'verification'` or `type: 'uat'`

**5. Remove all local file write instructions:**

Remove or replace any instructions that:
- Write SUMMARY.md files to `.planning/phases/` directories
- Read PLAN.md files from `.planning/phases/` directories
- Check for local SUMMARY.md existence to determine completion

**6. External edit detection (WIRE-06):**

Before reading a plan from GitHub, optionally call `mcp_detect_external_edits` to check if the phase issue was modified externally. If modified, warn the user before proceeding.
