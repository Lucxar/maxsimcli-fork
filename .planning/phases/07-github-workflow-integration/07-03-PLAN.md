<objective>
Rewrite the init workflows (new-project.md and init-existing.md) to mandate GitHub setup and automatically create phase issues on GitHub after roadmap creation.
</objective>

<execution_context>
Both init workflows are large (~1100 lines each, ~45KB). They orchestrate the full project initialization flow: codebase mapping, requirements gathering, roadmap creation. Phase 7 adds mandatory GitHub setup as a step, and phase issue creation after the roadmap is built.

Locked decisions:
- WIRE-03: Init REQUIRES GitHub setup. No GitHub remote = fail with instructions. No gh auth = fail with instructions. No local-only mode.
- WIRE-01: GitHub-first write order. Phase issues written to GitHub, not locally.
- WIRE-08: Phase issues start in "To Do" status on the board.

These are markdown workflow files (prompts for Claude), not TypeScript code. Changes are additive -- adding new steps/sections to existing flows.
</execution_context>

  <action>
Read the full `templates/workflows/new-project.md` file first.

Add a mandatory GitHub gate as an early step (after the init context is loaded but before any work begins). Insert a new section:

**1. GitHub Prerequisites Gate (add after init context loading, before codebase analysis):**

Add a section like:
```markdown
### GitHub Prerequisites Gate

**This gate is MANDATORY. Do not proceed if it fails.**

1. Check init context for `github_ready`, `has_github_remote`, `gh_authenticated`
2. If `has_github_remote` is false:
   - STOP. Tell user: "No GitHub remote found. Run `git remote add origin <url>` first."
   - Do NOT proceed with project setup.
3. If `gh_authenticated` is false:
   - STOP. Tell user: "GitHub CLI not authenticated. Run `gh auth login` first."
   - Do NOT proceed with project setup.
4. Call `mcp_github_setup` with the milestone title
5. If setup fails, STOP with the error message. Do not fall back to local-only mode.
6. Record the project_number and board details for use in later steps.
```

**2. Phase Issue Creation (add after roadmap creation step):**

After the step that creates/finalizes ROADMAP.md, add a new section:

```markdown
### Create Phase Issues on GitHub

After the roadmap is finalized, create a GitHub Issue for each phase:

1. Parse the roadmap to extract all phases (phase number, name, goal, requirements, success criteria)
2. For each phase, call `mcp_create_phase_issue` with:
   - phase_number, phase_name, goal, requirements, success_criteria
3. The tool auto-adds each issue to the project board with "To Do" status
4. If any phase issue creation fails:
   - Log which phases succeeded and which failed
   - Offer targeted retry for failed phases only (WIRE-07 rollback)
5. Report: "Created {N} phase issues on GitHub Project Board #{project_number}"
```

**3. Remove or gate any local artifact file creation steps:**

If the workflow writes any phase-level artifact files (PLAN.md, SUMMARY.md) to `.planning/phases/`, add a comment noting these are no longer created per WIRE-02. The roadmap itself (ROADMAP.md) stays local as it is project context, not a phase artifact.

Make sure NOT to remove: PROJECT.md creation, REQUIREMENTS.md creation, config.json creation, STATE.md creation, ROADMAP.md creation -- these all stay local per WIRE-02 scope boundaries.
