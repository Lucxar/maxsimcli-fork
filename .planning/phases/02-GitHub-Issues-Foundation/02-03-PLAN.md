<action>
**Complete rewrite of projects.ts** to use Octokit REST API. Delete all existing code and write from scratch.

Remove all imports from `./gh-legacy.js`. Import from `./client.js` instead.

**Required exports:**

1. `ensureProjectBoard(title: string)` -> `Promise<GhResult<{ projectNumber: number; projectId: string }>>`
   - First, detect whether the repo owner is a user or org. Use `octokit.rest.repos.get({ owner, repo })` and check `response.data.owner.type` ('User' or 'Organization').
   - List existing projects to check if one with the given title already exists.
   - For user-owned repos: `octokit.request('GET /users/{username}/projects', { username: owner })` OR use the Projects v2 REST endpoints if available.
   - If no matching project exists, create one.
   - After creation, ensure the project has a "Status" single-select field with options matching `DEFAULT_STATUS_OPTIONS` ('To Do', 'In Progress', 'In Review', 'Done').
   - Store the project number and status field details in the mapping cache.

   **IMPORTANT:** The Projects v2 REST API may not be fully available in all Octokit versions. If the REST endpoints return 404 or are unavailable, fall back to using `gh project create --owner @me --title "${title}" --format json` via a direct `execFileSync('gh', [...])` call as a compatibility bridge. Document this fallback clearly with a TODO for removal once Octokit REST support is confirmed.

2. `addItemToProject(projectNumber: number, issueNodeId: string)` -> `Promise<GhResult<{ itemId: string }>>`
   - Add an issue to the project board.
   - Try Projects v2 REST endpoint first, fall back to `gh project item-add` CLI if needed.

3. `moveItemToStatus(projectNumber: number, itemId: string, status: IssueStatus)` -> `Promise<GhResult<void>>`
   - Update the Status field of a project item to the given column.
   - Maps status string to the status option ID (from mapping cache or fresh lookup).

4. `getProjectBoard(projectNumber: number)` -> `Promise<GhResult<{ items: Array<{ id: string; issueNumber: number; status: IssueStatus }> }>>`
   - List all items in the project with their current status.
   - Use pagination for large projects.

**Handling Projects v2 REST API uncertainty:**

The RESEARCH.md notes that the Projects v2 REST API typed methods in Octokit are unverified (MEDIUM confidence). Strategy:

a) Try `octokit.request()` with explicit REST paths first (e.g., `GET /users/{username}/projectsV2/{project_number}/items`).

b) If the REST endpoint returns 404 or type errors at compile time, use `gh project` CLI commands as a bridge (via `execFileSync` from `node:child_process`). This is acceptable as a TEMPORARY measure because:
   - Project board operations are infrequent (only during init and status changes)
   - The overhead per `gh` spawn is negligible for these rare operations
   - This can be replaced with pure Octokit once typed methods are confirmed

c) Add a `// TODO(v5.1): Replace gh CLI bridge with Octokit REST when typed methods available` comment at each bridge point.

**What to avoid:**
- Do NOT use GraphQL (CONTEXT.md locks REST-only)
- Do NOT import from `./gh-legacy.js` -- if you need gh CLI for Projects v2 bridge, use direct `execFileSync('gh', [...])` calls with a clear comment explaining why
- Do NOT assume Projects v2 REST endpoints have typed Octokit methods -- use `octokit.request()` with explicit paths

<name>Rewrite sync.ts for GitHub-native state queries</name>
<files>

<action>
**Complete rewrite of sync.ts** to use Octokit for state verification. Delete all existing code and write from scratch.

Remove all imports from `./gh-legacy.js`. Import from `./client.js` instead.

The purpose of sync.ts changes from "sync local state with GitHub" to "read state FROM GitHub" (since GitHub IS the source of truth now).

**Required exports:**

1. `checkPhaseProgress(phaseIssueNumber: number)` -> `Promise<GhResult<PhaseProgress>>`

   Define `PhaseProgress`:
   ```typescript
   interface PhaseProgress {
     total: number;
     completed: number;
     inProgress: number;
     remaining: number;
     tasks: Array<{
       number: number;
       title: string;
       state: 'open' | 'closed';
     }>;
   }
   ```

   Implementation:
   - Use `octokit.rest.issues.listSubIssues({ owner, repo, issue_number: phaseIssueNumber })` with pagination
   - Count open vs closed sub-issues
   - Return structured progress

2. `getPhaseState(phaseIssueNumber: number)` -> `Promise<GhResult<'to-do' | 'in-progress' | 'in-review' | 'done'>>`
   - If all sub-issues are closed -> 'done'
   - If some sub-issues are closed and some open -> 'in-progress'
   - If no sub-issues exist or all are open -> 'to-do'
   - 'in-review' state is determined by project board column (check if caller needs this)
   - For simplicity in v1: derive state from sub-issue counts. Project board is the authoritative source for in-review, but this function provides a quick approximation.

3. `detectInterruptedPhase(phaseIssueNumber: number)` -> `Promise<GhResult<{ interrupted: boolean; completedTasks: number[]; remainingTasks: number[] }>>`
   - CONTEXT.md decision: "Issue state is truth -- mix of open/closed sub-issues indicates interrupted execution"
   - List sub-issues. If there's a mix of open and closed, the phase was interrupted.
   - Return the issue numbers of completed (closed) and remaining (open) tasks.
   - This function is called by the resume logic (ARCH-05).

4. `getAllPhasesProgress()` -> `Promise<GhResult<Array<{ phaseNumber: string; title: string; issueNumber: number; progress: PhaseProgress }>>>`
   - List all issues with label 'phase' using `octokit.paginate()`
   - For each phase, get its sub-issue progress
   - Parse phase number from title pattern `[Phase XX]`
   - Return sorted by phase number

**What to avoid:**
- Do NOT use GraphQL batch queries (old pattern in current sync.ts)
- Do NOT import from `./gh-legacy.js`
- Do NOT read from local files to determine state -- always query GitHub
- Do NOT use `Promise.all` for multiple API calls with mutations -- use sequential calls for safety
</action>
