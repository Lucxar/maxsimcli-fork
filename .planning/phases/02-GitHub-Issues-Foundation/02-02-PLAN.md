<objective>
Rewrite the issue, label, and milestone modules to use Octokit instead of gh CLI. After this plan:
1. Phase Issues are created with structured YAML frontmatter bodies and the 'phase' label
2. Task sub-issues are created and linked to parent phases via GitHub's native sub-issue API
3. Plans are posted as structured comments on phase Issues
4. Labels and milestones are managed via Octokit typed methods
5. The mapping file is a rebuildable cache, not the source of truth
</objective>

<execution_context>
Plan 01 established the Octokit client singleton (`client.ts`), auth gate (`requireAuth()`), and updated types. This plan now rewrites the CRUD modules that were previously using `ghExec()` to use the Octokit client directly.

Key constraints from CONTEXT.md (locked decisions):
- Phase = Parent Issue with sub-issues (GitHub's native sub-issue feature)
- Plan = Issue comment on the phase Issue (not separate Issues)
- Tasks = Sub-issues created upfront during planning
- Issue body = high-level summary with YAML frontmatter for machine-parseable metadata
- Comments = detail (research findings, plan breakdowns, execution logs, reviews)
- Best-effort atomicity with cleanup on failure
- Minimal labels: phase, task, blocker

CRITICAL pitfall from RESEARCH.md: The sub-issues API uses the issue's internal `id` (a large integer), NOT the human-readable `number`. When creating issues, capture BOTH `response.data.number` and `response.data.id`.
</execution_context>

<name>Rewrite issues.ts with Octokit and native sub-issues</name>
<files>

<action>
**Complete rewrite of issues.ts** to use Octokit. Delete all existing code and write from scratch.

Remove all imports from `./gh-legacy.js`. Import from `./client.js` instead.

**Required exports:**

1. `createPhaseIssue(phaseNum: string, phaseName: string, goal: string, requirements: string[], successCriteria: string[])` -> `Promise<GhResult<{ number: number; id: number }>>`
   - Creates a GitHub Issue with title `[Phase ${phaseNum}] ${phaseName}`
   - Body contains YAML frontmatter block (wrapped in ```yaml code fence) with: `phase_number`, `requirements` array, `dependencies` array, `status: 'to-do'`
   - Body also contains human-readable sections: `## Goal`, `## Requirements`, `## Success Criteria`
   - Label: `phase`
   - Returns both `number` (for display) and `id` (for sub-issue linking)

2. `createTaskSubIssue(phaseNum: string, taskId: string, title: string, body: string, parentIssueNumber: number)` -> `Promise<GhResult<{ number: number; id: number }>>`
   - Creates a GitHub Issue with title `[P${phaseNum}] ${title}`
   - Label: `task`
   - Body is passed as-is (the caller formats it)
   - After creation, links as sub-issue: `octokit.rest.issues.addSubIssue({ owner, repo, issue_number: parentIssueNumber, sub_issue_id: childIssue.data.id })`
   - CRITICAL: Use `child.data.id` (internal ID), NOT `child.data.number` for `sub_issue_id`
   - If sub-issue linking fails, attempt cleanup (close the orphan issue) and return error

3. `postPlanComment(phaseIssueNumber: number, planNumber: string, planContent: string)` -> `Promise<GhResult<{ commentId: number }>>`
   - Creates a comment on the phase Issue using `octokit.rest.issues.createComment`
   - Comment body: `## Plan ${planNumber}\n\n${planContent}\n\n---\n*Posted by MAXSIM*`

4. `closeIssue(issueNumber: number, reason?: string)` -> `Promise<GhResult<void>>`
   - Uses `octokit.rest.issues.update({ state: 'closed', state_reason: 'completed' })`
   - If reason is provided, adds a comment before closing

5. `reopenIssue(issueNumber: number)` -> `Promise<GhResult<void>>`
   - Uses `octokit.rest.issues.update({ state: 'open' })`

6. `getPhaseIssue(phaseIssueNumber: number)` -> `Promise<GhResult<{ number: number; id: number; title: string; state: string; body: string }>>`
   - Fetches a single issue by number

7. `listPhaseSubIssues(phaseIssueNumber: number)` -> `Promise<GhResult<Array<{ number: number; id: number; title: string; state: string }>>>`
   - Uses `octokit.rest.issues.listSubIssues({ owner, repo, issue_number: phaseIssueNumber })`
   - Returns all sub-issues with their state (open/closed)

8. `postCompletionComment(issueNumber: number, commitSha: string, filesChanged: string[])` -> `Promise<GhResult<{ commentId: number }>>`
   - Posts a structured completion comment with commit SHA and file list
   - This satisfies the CONTEXT.md decision: "Each completed task gets a completion comment"

All functions use `withGhResult()` from client.ts to wrap Octokit calls. This provides consistent GhResult<T> return types with error code mapping.

**Sequential creation for rate limit safety:** When creating multiple tasks in batch, use a simple `for` loop (not `Promise.all`). The throttling plugin handles rate limits, but sequential creation is safer for mutations (RESEARCH.md Pitfall 5).

**What to avoid:**
- Do NOT import from `./gh-legacy.js` or use `ghExec` / `ghGraphQL`
- Do NOT use `node_id` -- the sub-issues API uses numeric `id`, not GraphQL node IDs
- Do NOT try to parse issue numbers from URLs (old pattern) -- Octokit returns structured response objects

<action>
**Step 1: Rewrite labels.ts.**

Delete existing code. Write a single export:

`ensureLabels()` -> `Promise<GhResult<void>>`
- Import `getOctokit`, `getRepoInfo`, `withGhResult` from `./client.js`
- Import `MAXSIM_LABELS` from `./types.js`
- For each label in MAXSIM_LABELS (phase, task, blocker):
  - Try `octokit.rest.issues.getLabel({ owner, repo, name: label.name })`
  - If 404 (not found), create it: `octokit.rest.issues.createLabel({ owner, repo, name, color, description })`
  - If it exists but color/description differ, update it: `octokit.rest.issues.updateLabel()`
- Return `{ ok: true, data: undefined }` on success

**Step 2: Rewrite milestones.ts.**

Delete existing code. Write:

`ensureMilestone(title: string, description?: string)` -> `Promise<GhResult<{ number: number }>>`
- List existing milestones: `octokit.rest.issues.listMilestones({ owner, repo, state: 'all' })`
- If one with matching title exists, return its number
- If not, create it: `octokit.rest.issues.createMilestone({ owner, repo, title, description })`
- Return the milestone number

`closeMilestone(milestoneNumber: number)` -> `Promise<GhResult<void>>`
- Update milestone state to 'closed': `octokit.rest.issues.updateMilestone({ owner, repo, milestone_number, state: 'closed' })`

**Step 3: Update mapping.ts to be a rebuildable cache.**

The current mapping.ts reads/writes `.planning/github-issues.json`. This file transitions from "source of truth" to "rebuildable performance cache". Make these changes:

a) Keep `loadMapping()` and `saveMapping()` -- they still provide fast local lookups.

b) Add `rebuildMappingFromGitHub()` -> `Promise<GhResult<IssueMappingFile>>`:
   - List all Issues with label 'phase' (using Octokit paginate)
   - For each phase issue, list its sub-issues
   - Reconstruct the mapping from GitHub state
   - Save to disk and return
   - This function ensures the cache can always be rebuilt from the source of truth (GitHub)

c) Add a comment at the top of the file documenting that this file is a CACHE, not the source of truth:
```typescript
/**
 * GitHub Issue Mapping — Local performance cache
 *
 * IMPORTANT: This file (.planning/github-issues.json) is a CACHE that can be
 * rebuilt from GitHub Issues at any time via rebuildMappingFromGitHub().
 * GitHub Issues are the single source of truth (ARCH-01).
 */
```

d) Update the mapping type to also store `id` (internal numeric ID) alongside `number` and `node_id` in TaskIssueMapping. The `id` field is needed for sub-issue operations.

**What to avoid:**
- Do NOT import from `./gh-legacy.js` in any of these files
- Do NOT use GraphQL for any operation
- Do NOT remove loadMapping/saveMapping -- the cache is still valuable for performance
