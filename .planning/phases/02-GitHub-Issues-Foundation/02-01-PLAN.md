<objective>
Set up the Octokit adapter foundation and enforce local-only installation. After this plan:
1. A new `client.ts` module provides a configured Octokit singleton with throttling/retry
2. Auth is a hard gate -- no `detectGitHubMode()` / `'local-only'` fallback
3. @octokit packages are installed and bundle correctly via tsdown into CJS
4. The installer only supports local installation (--global is removed)
</objective>

<execution_context>
This is Phase 2 (GitHub Issues Foundation) of the MAXSIM v5.0 roadmap. Phase 1 (Infrastructure Cleanup) is complete -- the codebase is lean, async-only, and builds cleanly. This plan establishes the Octokit adapter that all subsequent plans depend on.

Key constraints from CONTEXT.md (locked decisions):
- gh for auth (`gh auth token`), @octokit/rest for all API calls (REST, not GraphQL)
- Cache token in memory -- fetch once per CLI process, no disk persistence
- Hard failure when gh not installed/authenticated (ARCH-03)
- Local-only install to .claude/ per project (ARCH-04)
- Two-layer architecture: adapter (API calls) + domain (business logic)

The existing `src/github/gh.ts` wraps `gh` CLI via execFile. This plan REPLACES it with `client.ts` using @octokit/rest. The old `gh.ts` will be kept temporarily as `gh-legacy.ts` so that existing modules (issues.ts, projects.ts, etc.) still compile. Plans 02 and 03 will then rewrite those modules to use the new client, after which `gh-legacy.ts` gets deleted.
</execution_context>

```typescript
export class AuthError extends Error {
  constructor(
    public readonly code: 'NOT_INSTALLED' | 'NOT_AUTHENTICATED' | 'SCOPE_MISSING',
    message: string,
  ) {
    super(message);
    this.name = 'AuthError';
  }
}
```

c) Replace the `MAXSIM_LABELS` array with the new minimal label set (CONTEXT.md decision: `phase`, `task`, `blocker` only):
```typescript
export const MAXSIM_LABELS = [
  { name: 'phase', color: '6f42c1', description: 'MAXSIM phase issue' },
  { name: 'task', color: '0075ca', description: 'MAXSIM task issue' },
  { name: 'blocker', color: 'd73a4a', description: 'Blocker issue' },
] as const;
```

d) Keep everything else (GhResult, GhErrorCode, AuthStatus, IssueStatus, TaskIssueMapping, PhaseMapping, IssueMappingFile, FIBONACCI_POINTS, DEFAULT_STATUS_OPTIONS).

**Step 2: Create client.ts.**

Create `packages/cli/src/github/client.ts` with these exports:

1. `getOctokit()` -- Returns a singleton Octokit instance configured with throttling + retry plugins. Uses `gh auth token` (via `execFileSync`) to obtain the token on first call. Caches the instance in a module-level variable. The throttle config should retry twice on primary rate limits and once on secondary rate limits.

2. `requireAuth()` -- Synchronous check that throws `AuthError` on failure. Three checks in order:
   - Is `gh` CLI installed? (try `execFileSync('gh', ['--version'])`)
   - Is gh authenticated? (try `execFileSync('gh', ['auth', 'token'])` and verify non-empty)
   - Has the token got `project` scope? (try `execFileSync('gh', ['auth', 'status'])` and parse for 'project' scope)
   If any check fails, throw AuthError with the appropriate code and a helpful message.

3. `getRepoInfo()` -- Async function returning `{ owner: string; repo: string }`. Parses from `git remote get-url origin`. Caches the result in a module-level variable. Supports both HTTPS (`https://github.com/owner/repo.git`) and SSH (`git@github.com:owner/repo.git`) remote URLs.

4. `resetOctokit()` -- Resets the singleton (for testing). Sets `_instance = null` and `_cachedRepo = null`.

5. `withGhResult<T>(fn: () => Promise<T>): Promise<GhResult<T>>` -- Error wrapper that catches Octokit `RequestError` and maps HTTP status codes to `GhErrorCode`. Import `RequestError` from `@octokit/request-error` (it's a transitive dependency of @octokit/rest).

Use the exact patterns from RESEARCH.md (Pattern 1 and Example 4) but with these adjustments:
- Import `execFileSync` from `node:child_process` (not the async version -- token fetch is synchronous and happens once)
- Set `userAgent: 'maxsimcli'`
- The `retry` plugin option is just `{ enabled: true }` (the @octokit/plugin-retry handles the rest)

**Step 3: Rename gh.ts to gh-legacy.ts.**

Rename `packages/cli/src/github/gh.ts` to `packages/cli/src/github/gh-legacy.ts`. This keeps existing modules (issues.ts, projects.ts, sync.ts, labels.ts, milestones.ts) compiling since they import from `./gh.js`. Update their imports from `'./gh.js'` to `'./gh-legacy.js'` in each file. This is a temporary bridge -- Plans 02 and 03 will rewrite these modules to use client.ts and then gh-legacy.ts will be deleted.

Files to update imports in: issues.ts, projects.ts, sync.ts, labels.ts, milestones.ts.

**Step 4: Update index.ts barrel export.**

Replace `export * from './gh.js'` with:
```typescript

export * from './gh-legacy.js';
```

Add `client.js` export. Keep `gh-legacy.js` export so downstream code still compiles.

**What to avoid:**
- Do NOT delete gh.ts outright -- rename to gh-legacy.ts so existing imports work after find-replace
- Do NOT use async `execFile` for token fetching -- use `execFileSync` since it only happens once and simplifies the singleton pattern
- Do NOT use GraphQL anywhere (CONTEXT.md locks REST-only)
