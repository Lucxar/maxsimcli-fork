<objective>
Build the MCP tool layer that exposes GitHub operations to Claude Code, the sync check mechanism, and wire everything into the MCP server. This replaces existing local-only task tracking with GitHub-backed implementations, adds new board query tools, and provides PR creation with auto-close linking (AC-08).
</objective>

<execution_context>
Plans 01-03 built the GitHub module layer (gh.ts, types.ts, mapping.ts, projects.ts, labels.ts, milestones.ts, templates.ts, issues.ts). This plan builds the MCP tools on top and creates the sync mechanism.

Key constraint on tool coexistence (Decision #8 clarification): Decision #8 says "full replacement" — this means the AI PREFERS GitHub tools for task tracking when in 'full' mode. Existing phase-tools.ts and todo-tools.ts are NOT removed because they handle non-GitHub operations (local phase CRUD, state management) and serve as the degradation fallback in 'local-only' mode. The new github-tools.ts and board-tools.ts add NEW MCP tools that the AI uses as the primary task-tracking interface.

Key constraint: Graceful degradation [Decision #13, NO-GO]. Every GitHub MCP tool must check `detectGitHubMode()`. In `local-only` mode, fall back to existing local behavior (existing tool functions still work). In `full` mode, perform GitHub operations AND local operations.

AC-08: PRs auto-link via `Closes #N` — requires `buildPrBody()` (from issues.ts, Plan 03) to be callable via an MCP tool so the AI can generate PR descriptions with auto-close links.
AC-09: Sync check before each phase action detects external changes.
AC-10: All existing task-tracking MCP tools replaced with GitHub-backed implementations.
AC-11: New MCP tools: query board, search/filter issues, post comment, move cards.
</execution_context>

<action>
Create the sync check module and barrel export.

**sync.ts** exports:

1. **syncCheck(cwd: string): Promise&lt;GhResult&lt;{ inSync: boolean; changes: Array&lt;{ issueNumber: number; field: string; localValue: string; remoteValue: string }&gt; }&gt;&gt;**

   Compare local mapping file against GitHub reality:
   a. Load mapping from `github-issues.json`.
   b. For each tracked issue, run `gh issue view {number} --json state,labels,title` to get current GitHub state.
   c. Compare: if an issue was closed externally (state=closed but local mapping shows "In Progress"), flag it as a change.
   d. Return list of discrepancies.
   e. For efficiency, batch issue lookups: use `gh api graphql` with a single query that fetches multiple issues at once (up to 100 per query). This avoids N+1 sequential calls.

2. **verifyIssueState(issueNumber: number, expectedState: 'open' | 'closed'): Promise&lt;GhResult&lt;{ verified: boolean; actualState: string }&gt;&gt;** — Quick single-issue state check.

3. **handleExternalClose(cwd: string, issueNumber: number): Promise&lt;GhResult&lt;{ action: 'accepted' | 'reopened'; reason: string }&gt;&gt;** — When sync detects an externally closed issue: return info for the AI to decide whether to accept (code actually implements the task) or reopen (code doesn't match). The AI makes the decision based on code verification — this function just provides the data.

<action>
Create the MCP tool files and wire them into the MCP server.

**github-tools.ts** — `registerGitHubTools(server: McpServer)`:

Follow the EXACT pattern from existing tools (see phase-tools.ts): use `server.tool()`, zod schemas, async handler, try/catch with mcpError, detectProjectRoot(). Every tool checks `detectGitHubMode()` first and degrades gracefully.

Tools to register:

1. **mcp_github_setup** — "Set up GitHub integration: create project board, labels, milestone, and issue templates."
   Params: `{ milestone_title: string }` (optional, defaults to current milestone from STATE.md)
   Action: Call ensureProjectBoard, ensureLabels, ensureMilestone, installIssueTemplates. Store results in mapping file. Return setup summary.
   Degradation: In local-only mode, only install issue templates (those are local files).

2. **mcp_create_plan_issues** — "Create GitHub issues for all tasks in a finalized plan. Creates task issues and parent tracking issue."
   Params: `{ phase: string, plan: string, tasks: array of task objects, milestone?: string }`
   Action: Call createAllPlanIssues. Add all issues to project board. Set estimates. Move all to "To Do" status.
   Degradation: In local-only mode, return success with warning "GitHub not configured, issues not created".

3. **mcp_create_todo_issue** — "Create a GitHub issue for a todo item."
   Params: `{ title: string, description?: string, acceptance_criteria?: string[] }`
   Action: Call createTodoIssue. Add to project. Update mapping.
   Degradation: Fall through to existing local todo creation.

4. **mcp_move_issue** — "Move a GitHub issue to a new status column (To Do, In Progress, In Review, Done)."
   Params: `{ issue_number: number, status: 'To Do' | 'In Progress' | 'In Review' | 'Done' }`
   Action: Load mapping for item_id and field IDs. Call moveItemToStatus. Update local mapping.
   Degradation: In local-only mode, just update local mapping status field.

5. **mcp_close_issue** — "Close a GitHub issue as completed or not planned."
   Params: `{ issue_number: number, reason?: 'completed' | 'not_planned' }`
   Action: Call closeIssue. Move to Done on board. Update mapping.

6. **mcp_post_comment** — "Post a progress comment on a GitHub issue."
   Params: `{ issue_number: number, body: string }`
   Action: Call postComment.

7. **mcp_import_issue** — "Import an external GitHub issue into MAXSIM tracking."
   Params: `{ issue_number: number }`
   Action: Call importExternalIssue. Add maxsim + imported labels. Return issue details for AI to decide placement.

8. **mcp_sync_check** — "Check for external changes to tracked GitHub issues."
   Params: none
   Action: Call syncCheck. Return list of discrepancies.

9. **mcp_supersede_plan** — "Close old plan issues and link to new plan issues."
   Params: `{ phase: string, old_plan: string, new_plan: string }`
   Action: Call supersedePlanIssues.

10. **mcp_create_pr** — "Create a pull request with auto-close linking for tracked GitHub issues. Generates PR description with Closes #N lines (AC-08)."
    Params: `{ issue_numbers: number[], branch: string, title: string, base?: string, additional_context?: string, draft?: boolean }`
    Action: Call `buildPrBody(issue_numbers, additional_context)` from issues.ts to generate the PR body with `Closes #N` lines. Then run `gh pr create --title "..." --body "..." --head {branch} [--base {base}] [--draft]`. Parse the PR URL from stdout. Return `{ pr_number, pr_url, issues_linked }`.
    Degradation: In local-only mode, return the generated PR body text so the user can create the PR manually, but skip the `gh pr create` call.
    Note: This tool is the critical wiring that makes AC-08 (PRs auto-link via Closes #N) work end-to-end. Without it, `buildPrBody()` in issues.ts would be orphaned code.

**board-tools.ts** — `registerBoardTools(server: McpServer)`:

1. **mcp_query_board** — "Query the GitHub project board. Returns all items with their status, estimates, and issue details."
   Params: `{ status?: 'To Do' | 'In Progress' | 'In Review' | 'Done', phase?: string }`
   Action: Use `gh project item-list {num} --owner @me --format json`. Filter by status and/or phase if provided. Return structured list.
   Degradation: In local-only mode, read mapping file and return local status.

2. **mcp_search_issues** — "Search GitHub issues by label, milestone, state, or text query."
   Params: `{ labels?: string[], milestone?: string, state?: 'open' | 'closed' | 'all', query?: string }`
   Action: Use `gh issue list --json number,title,state,labels,milestone --label ... --milestone ... --search ...`. Return filtered results.
   Degradation: In local-only mode, return local mapping data.

3. **mcp_get_issue_detail** — "Get full details of a specific GitHub issue including comments."
   Params: `{ issue_number: number }`
   Action: Use `gh issue view {number} --json number,title,body,state,labels,comments,assignees`.

4. **mcp_set_estimate** — "Set Fibonacci story points on a GitHub issue."
   Params: `{ issue_number: number, points: number }`
   Action: Validate points is in FIBONACCI_POINTS. Load mapping for item_id. Call setEstimate.
