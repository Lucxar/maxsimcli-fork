<objective>
Build the GitHub setup infrastructure: project board creation with 4-column kanban, label management, milestone CRUD, and issue template installation. These modules enable the one-time setup flow that creates all GitHub artifacts needed before issues can be created.
</objective>

<execution_context>
Plan 01 created the foundation (gh.ts, types.ts, mapping.ts). This plan builds the setup layer on top.

Key constraints from locked decisions:
- One Project board per repo (not per milestone) [Decision #9]
- 4 columns: To Do, In Progress, In Review, Done [Decision #10] -- GitHub defaults only have 3, must add "In Review"
- Fibonacci story points via Estimate number field [Decision #11]
- Labels created upfront during setup [from CONTEXT.md]
- Issue templates in .github/ISSUE_TEMPLATE/ [from CONTEXT.md]
- Milestones used for grouping (one per MAXSIM milestone) [Decision #9]

Key pitfalls from research:
- Projects v2 defaults only create "Todo", "In Progress", "Done" -- must add "In Review" via GraphQL mutation
- The `project` scope is required for Projects v2 commands -- check and warn
- `gh project item-add` gives an item_id separate from issue node_id -- store both
</execution_context>

<action>
Create the project board setup and label management modules.

**projects.ts** exports:

1. **createProjectBoard(title: string): Promise&lt;GhResult&lt;{ number: number; id: string }&gt;&gt;** — Run `gh project create --owner @me --title "{title}" --format json`. Parse JSON response for project number and id.

2. **ensureProjectBoard(title: string, cwd: string): Promise&lt;GhResult&lt;{ number: number; id: string; created: boolean }&gt;&gt;** — Check mapping file for existing project. If found, verify it still exists via `gh project view`. If not found, call createProjectBoard, then set up fields and status options. Save to mapping. Return with `created: true/false`.

3. **getProjectFields(projectNum: number): Promise&lt;GhResult&lt;Array&lt;{ id: string; name: string; type: string; options?: Array&lt;{ id: string; name: string }&gt; }&gt;&gt;&gt;** — Run `gh project field-list {num} --owner @me --format json`. Parse and return field list.

4. **addStatusOption(projectId: string, statusFieldId: string, optionName: string, existingOptions: Array&lt;{ id: string; name: string }&gt;): Promise&lt;GhResult&lt;string&gt;&gt;** — Use `ghGraphQL` with `updateProjectV2Field` mutation to add a new single-select option. Include ALL existing options plus the new one (the mutation replaces all options). Return the new option's ID. See RESEARCH.md Example 6 for the exact mutation shape.

5. **setupProjectFields(projectNum: number, projectId: string, cwd: string): Promise&lt;GhResult&lt;void&gt;&gt;** — Orchestrates field setup: (a) get existing fields, (b) find Status field, verify "In Review" option exists or add it, (c) create Estimate NUMBER field via `gh project field-create`, (d) store all field/option IDs in mapping file.

6. **addItemToProject(projectNum: number, issueUrl: string): Promise&lt;GhResult&lt;{ item_id: string }&gt;&gt;** — Run `gh project item-add {num} --owner @me --url {issueUrl} --format json`. Return the project item ID.

7. **moveItemToStatus(projectId: string, itemId: string, statusFieldId: string, statusOptionId: string): Promise&lt;GhResult&lt;void&gt;&gt;** — Run `gh project item-edit` with the correct flags.

8. **setEstimate(projectId: string, itemId: string, estimateFieldId: string, points: number): Promise&lt;GhResult&lt;void&gt;&gt;** — Run `gh project item-edit` with `--number` flag.

**labels.ts** exports:

1. **ensureLabels(): Promise&lt;GhResult&lt;void&gt;&gt;** — Iterate over MAXSIM_LABELS (imported from types.ts), run `gh label create {name} --color {color} --description {desc} --force` for each. The `--force` flag updates existing labels. Continue on individual label failures (log to stderr), only fail if ALL labels fail.

Import `ghExec`, `ghGraphQL` from `./gh.js` and types from `./types.js`.
</action>

<action>
Create milestone management and issue template installation modules.

**milestones.ts** exports:

1. **createMilestone(title: string, description?: string): Promise&lt;GhResult&lt;{ number: number; id: number }&gt;&gt;** — Use `gh api repos/{owner}/{repo}/milestones -X POST -f title=... -f description=... -f state=open`. Parse JSON response for milestone number. Note: use `{owner}/{repo}` placeholder syntax that gh CLI auto-resolves.

2. **findMilestone(title: string): Promise&lt;GhResult&lt;{ number: number; id: number } | null&gt;&gt;** — Use `gh api repos/{owner}/{repo}/milestones --jq '.[] | select(.title == "TITLE")'`. Return null if not found.

3. **ensureMilestone(title: string, description?: string): Promise&lt;GhResult&lt;{ number: number; id: number; created: boolean }&gt;&gt;** — Find existing or create new. Idempotent.

4. **closeMilestoneIfComplete(milestoneNumber: number): Promise&lt;GhResult&lt;{ closed: boolean }&gt;&gt;** — Get milestone details, check if open_issues === 0, if so PATCH state to "closed". This implements AC-12 (milestones auto-close when all issues closed).

**templates.ts** exports:

1. **installIssueTemplates(cwd: string): void** — Write two YAML issue template files to `{cwd}/.github/ISSUE_TEMPLATE/`:

   a. `phase-task.yml` — Name: "MAXSIM Phase Task", labels: ["maxsim", "phase-task"], body with: markdown intro ("Auto-generated by MAXSIM"), textarea for Summary (required), textarea for Full Specification.

   b. `todo.yml` — Name: "MAXSIM Todo", labels: ["maxsim", "todo"], body with: textarea for Description (required), textarea for Acceptance Criteria.

   Create `.github/ISSUE_TEMPLATE/` directory recursively if it doesn't exist. Use `fs.mkdirSync` with `recursive: true`. This is a synchronous file write operation (no gh CLI needed). See RESEARCH.md Example 7 for the exact YAML format.
</action>
