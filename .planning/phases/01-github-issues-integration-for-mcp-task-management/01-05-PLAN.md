<objective>
Wire GitHub operations into the EXISTING MCP phase, todo, and state tools so that every lifecycle event (create, complete, block) also updates GitHub. This is the "replacement" integration — existing tools gain GitHub-backed behavior when in 'full' mode while preserving local-only fallback.
</objective>

<execution_context>
Plans 01-04 built the GitHub module layer and new MCP tools. This plan MODIFIES the existing MCP tools (phase-tools.ts, todo-tools.ts, state-tools.ts) to also trigger GitHub operations when `detectGitHubMode()` returns `'full'`.

The modification pattern: after each existing local operation succeeds, check GitHub mode and perform the corresponding GitHub operation. If the GitHub operation fails, log warning but do NOT fail the local operation (local is always the fallback).

Key constraints:
- Existing tool interfaces (params, return types) MUST NOT change — backward compatibility
- GitHub operations are additive (local operation + GitHub operation), not replacement
- Sync check runs before phase actions (AC-09)
- Reviewer bounce: moving issue back to In Progress with comment (AC-05)
- Graceful degradation: if GitHub call fails, warn but succeed locally (AC-17)
</execution_context>

<action>
Modify the existing phase-tools.ts to add GitHub integration. Read the current file first and add GitHub hooks to existing tools.

**Changes to mcp_find_phase:**
- After finding phase locally, also check mapping file for GitHub issue numbers. Add `github_tracking_issue` and `github_task_issues` to the response data if available.

**Changes to mcp_create_phase:**
- After creating phase locally: if in 'full' mode, no additional GitHub action needed here (issues are created on plan finalization, not phase creation). This is per the locked decision of eager creation on plan finalization.

**Changes to mcp_complete_phase:**
- BEFORE completing locally: run `syncCheck(cwd)` and include any discrepancies in the response.
- AFTER completing locally: if in 'full' mode, close the parent tracking issue for this phase. Move it to "Done" on the board. Check if all issues in the milestone are closed — if so, call `closeMilestoneIfComplete()`.

**Changes to mcp_list_phases:**
- After listing locally: if in 'full' mode and mapping exists, enrich each phase entry with GitHub issue counts (open/closed).

**New tool: mcp_bounce_issue** (add to this file):
- "Bounce a task back from In Review to In Progress with a detailed comment explaining what failed."
- Params: `{ issue_number: number, reason: string }`
- Action: Move issue to "In Progress" on board. Post comment with the reviewer's feedback. This implements AC-05 (reviewer can bounce issues back to In Progress with detailed comment).
- Degradation: In local-only mode, just return success with a note that the bounce was recorded locally.

Import `detectGitHubMode`, `syncCheck`, `loadMapping`, `moveItemToStatus`, `closeIssue`, `postComment`, `closeMilestoneIfComplete` from `../github/index.js`.

For EVERY GitHub call: wrap in try/catch. If it fails, add a `github_warning` field to the response but do NOT fail the overall tool call. The local operation result is always the primary response.

Pattern for degradation:
```typescript
const mode = await detectGitHubMode();
if (mode === 'full') {
  try {
    // GitHub operations
  } catch (e) {
    // Add warning to response, do not fail
    githubWarning = `GitHub operation failed: ${(e as Error).message}`;
  }
}
```
