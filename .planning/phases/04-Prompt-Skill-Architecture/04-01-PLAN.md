**Skill specifications (from CONTEXT.md Section 4):**

1. **handoff-contract** (Protocol, ~30 lines):
   - Structured return format every agent MUST use: Key Decisions, Artifacts, Status, Deferred Items
   - Include the exact markdown template from AGENTS.md handoff contract section
   - Include guidance on when to use each status: complete, blocked, partial
   - Description: "Structured return format for agent handoffs. Defines Key Decisions, Artifacts, Status, and Deferred Items sections. Use when completing any agent task, returning results to orchestrator, or transitioning between workflow stages."

2. **verification-gates** (Protocol, ~100 lines):
   - Full gate framework: 4 gate types (input validation, pre-action, completion, quality)
   - Each gate type with: when to trigger, what evidence is required, failure behavior
   - Retry protocol: 2 retries (3 total attempts), feedback includes what failed + expected evidence + actual result
   - Escalation protocol: after 3rd failure, return full failure context to orchestrator
   - Anti-rationalization language: include ALL forbidden phrases from CONTEXT.md
   - Audit trail: log every gate attempt to GitHub Issues
   - Evidence block format: CLAIM / EVIDENCE / OUTPUT / VERDICT
   - Description: "Hard gate framework for evidence-based verification. Defines four gate types (input validation, pre-action, completion, quality), retry logic with feedback, anti-rationalization enforcement, and escalation protocol. Use when implementing verification checkpoints, completion gates, or quality checks."

3. **input-validation** (Protocol, ~25 lines):
   - Standard pattern for agent startup validation
   - Check required inputs exist (files, env vars, CLI args)
   - Fail immediately with structured error: AGENT RESULT: INPUT VALIDATION FAILED / Missing: [list] / Expected from: [source]
   - Do NOT attempt partial work on missing inputs
   - Description: "Validates required inputs exist before agent work begins. Checks file paths, environment variables, and CLI arguments. Use at agent startup to fail fast with structured error instead of proceeding with missing context."

4. **evidence-collection** (Methodology, ~60 lines):
   - How to gather fresh evidence using tool output
   - What counts as evidence table: Claim | Requires | Not Sufficient (expand from verification-before-completion skill)
   - Process: identify command, run fresh in THIS turn, read full output, check exit code, cite specific output
   - Common pitfalls table: Excuse | Why It Fails
   - Key rule: evidence must be from THIS turn (not prior turns, not assumptions)
   - Description: "Systematic evidence gathering using tool output before making claims. Covers what counts as evidence, collection process, and common pitfalls. Use when verifying work completion, checking test results, validating build success, or before any completion gate."

5. **research-methodology** (Methodology, ~80 lines):
   - Research process: define questions, identify sources, evaluate confidence
   - Tool priority for research: WebFetch for docs > Grep/Glob for codebase > Bash for CLI tools > WebFetch for blogs
   - Confidence levels: HIGH (official docs, verified), MEDIUM (reputable blogs, community consensus), LOW (single source, outdated)
   - Source evaluation: prefer primary sources, check dates, cross-reference claims
   - Output format: structured findings with confidence tags, sources section, open questions
   - Description: "Research process with tool priorities, confidence levels, and source evaluation. Defines how to investigate technical domains, evaluate sources, and structure findings. Use when researching libraries, APIs, architecture patterns, or any domain requiring external knowledge."

6. **agent-system-map** (Reference, ~40 lines):
   - Registry of all 4 agents: executor, planner, researcher, verifier
   - For each agent: role, base tools, preloaded skills, typical tasks
   - Orchestrator-mediated pattern: agents return results to orchestrator, orchestrator spawns next agent (subagents CANNOT spawn subagents -- this is a Claude Code limitation)
   - Spawn prompt format: natural language with ## Task, ## Context, ## Files to Read, ## Suggested Skills, ## Success Criteria
   - Description: "Registry of MAXSIM's 4 agent types with roles, tools, and skill mappings. Documents orchestrator-mediated communication pattern. Use when spawning agents, understanding agent capabilities, or designing agent interactions."

7. **commit-conventions** (Convention, ~25 lines):
   - Conventional commits: fix:, feat:, chore:, docs:, test:, refactor: with scope
   - Atomic commits: one logical change per commit
   - Breaking changes: fix!: / feat!: for breaking changes
   - Scope format: fix(phase-04): description
   - Co-author line for Claude: Co-Authored-By: Claude <noreply@anthropic.com>
   - Description: "Commit message format using conventional commits with scope. Defines atomic commit rules, breaking change markers, and co-author attribution. Use when creating git commits, reviewing commit messages, or establishing commit conventions for a project."

8. **tool-priority-guide** (Reference, ~40 lines):
   - When to use Read vs Bash for file reading (Read tool preferred)
   - When to use Grep vs Bash grep (Grep tool preferred)
   - When to use Glob vs Bash find (Glob tool preferred)
   - When to use Write vs Bash echo (Write tool preferred)
   - When to use Edit vs sed/awk (Edit tool preferred)
   - When to use WebFetch vs curl (WebFetch preferred)
   - General rule: dedicated tools over Bash equivalents for better UX and permissions
   - Description: "Tool selection guide for Claude Code operations. Maps common tasks to preferred tools (Read over cat, Grep over rg, Glob over find, Write over echo). Use when deciding which tool to use for file operations, search, or web content."
