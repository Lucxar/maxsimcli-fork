- No consistent structure for cross-skill suggestions

CONTEXT.md decisions to honor:
- Descriptions match existing MAXSIM skill description patterns (keyword-rich, action-oriented)
- Skills can mention "see also: /related-skill" but do NOT auto-load sub-skills
- Rewrite from scratch (not incremental migration)

- Varied size -- match content to scope

**1. using-maxsim (rewrite, ~60 lines):**
- Remove `alwaysApply: true` -- always-on content now lives in `.claude/rules/`
- Update to reference the 4-agent model: executor, planner, researcher, verifier
- Update to reference the 9-command surface: init, plan, execute, progress, go, debug, quick, settings, help
- Keep the routing table structure but update all references
- Mention skills can be loaded on-demand via description matching or `/skill-name` invocation
- Remove the "Available Agents" table that listed 13 agents -- replace with 4
- Description: "Routes work through MAXSIM's spec-driven workflow: checks planning state, determines active phase, dispatches to the correct MAXSIM command. Use when starting work sessions, resuming work, or choosing which MAXSIM command to run."

**2. verification-before-completion (rewrite, ~50 lines):**
- Keep the core evidence-before-claims philosophy
- Remove content now covered by rules/verification-protocol.md (the always-on part)
- Focus on the METHODOLOGY: the 5-step process (Identify, Run, Read, Verify, Claim)
- Keep evidence block format, "What Counts as Verification" table, and common pitfalls
- Add "See also: /verification-gates" for the full gate framework
- Description: "Requires running verification commands and reading actual output before completion claims. Covers the 5-step verification process and evidence block format. Use when claiming work is done, tests pass, builds succeed, or bugs are fixed."

**3. tdd (rewrite, ~50 lines):**
- Red-green-refactor cycle with atomic commits at each stage
- When to use TDD: business logic with defined I/O, API endpoints with contracts, data transformations, validation rules, algorithms, state machines
- When NOT to use TDD: UI layout, configuration, build scripts, one-off scripts
- Each TDD cycle produces 2-3 commits: red (failing test), green (minimal code to pass), refactor (cleanup)
- Context budget: TDD uses ~40% context due to RED-GREEN-REFACTOR overhead
- Description: "Test-driven development with red-green-refactor cycle and atomic commits. Write failing test first, then minimal passing code, then refactor. Use when implementing business logic, API endpoints, data transformations, validation rules, or algorithms."

**4. systematic-debugging (rewrite, ~50 lines):**
- 5-step process: Reproduce, Hypothesize, Isolate, Verify, Fix
- Each step requires evidence (tool output) before proceeding
- Hypothesis testing: form hypothesis, design test, run test, evaluate
- Do not guess-and-fix: always reproduce first, always verify the fix
- Description: "Systematic debugging via reproduce-hypothesize-isolate-verify-fix cycle. Requires evidence at each step. Use when investigating bugs, test failures, unexpected behavior, or runtime errors."

**5. code-review (rewrite, ~60 lines):**
- Review dimensions: security, public interfaces, error handling, test coverage, conventions compliance
- For each dimension: what to check, what evidence to gather, common findings
- Structured output: finding with severity (critical, major, minor), evidence, recommendation
- Distinguish spec-review (does it match requirements?) from code-review (is it correct and quality?)
- The verifier agent uses this skill for code review tasks (orchestrator directs via spawn prompt)
- Description: "Code quality review covering security, interfaces, error handling, test coverage, and conventions. Produces structured findings with severity and evidence. Use when reviewing pull requests, completed implementations, or code changes."

**6. sdd (rewrite, ~60 lines):**
- Spec-driven development: each task gets a spec (PLAN.md), fresh executor agent, verified completion
- Why: prevents context rot by giving each task a fresh context window
- Orchestrator pattern: read plan, spawn executor per task (or per plan), collect results
- Verification: spec-compliance review then code-quality review, retry until both pass
- When to use: sequential tasks where context accumulation causes quality degradation
- Description: "Spec-driven development with fresh-agent-per-task execution. Prevents context rot by isolating each task in a clean context window with its spec. Use when executing multi-task plans, orchestrating agent work, or when context accumulation degrades quality."

**1. brainstorming (rewrite, ~40 lines):**
- Multi-approach exploration before design decisions
- Process: define problem, generate 3+ approaches, evaluate tradeoffs, select with justification
- Output format: problem statement, approaches table (approach | pros | cons), selected approach with rationale
- When to use: before architectural decisions, before choosing between libraries/tools, before design

**2. roadmap-writing (rewrite, ~70 lines):**
- Phased planning with success criteria
- Structure: phases with goal, dependencies, requirements, success criteria
- Success criteria must be observable truths (verifiable by human using the application)
- Dependency graph: phases form a DAG, each phase declares what it depends on
- Requirement mapping: every requirement ID must appear in at least one phase
- Coverage map: requirement -> phase assignment
- Description: "Phased planning with dependency graphs, success criteria, and requirement mapping. Produces roadmaps with observable truths as success criteria. Use when creating project roadmaps, breaking features into phases, or structuring multi-phase work."

**3. maxsim-batch (rewrite, ~40 lines):**
- Parallel worktree execution: isolate agents in separate git worktrees
- Decomposition: split work into independent units (no shared file conflicts)
- Coordination: orchestrator spawns, tracks, and collects results from parallel agents
- Merge strategy: fast-forward preferred, rebase if needed, conflict resolution
- Limits: up to 30 parallel agents, but typically 3-10 for manageable coordination
- Description: "Parallel worktree execution for independent work units. Isolates agents in separate git worktrees for conflict-free parallel implementation. Use when executing multiple independent plans, batch processing, or parallelizable tasks."

**4. maxsim-simplify (rewrite, ~50 lines):**
- Maintainability optimization pass
- Dimensions: duplication (DRY), dead code removal, complexity reduction, naming clarity
- For each dimension: what to look for, how to measure, when to act
- Output: structured findings with before/after metrics
- Constraint: only simplify touched files unless explicitly asked for broader refactoring
- Description: "Maintainability optimization covering duplication, dead code, complexity, and naming. Produces structured findings with before/after metrics. Use when reviewing code for simplification, during refactoring passes, or when codebase complexity is increasing."

**5. memory-management (rewrite, ~40 lines):**
- Pattern and error persistence across sessions
- What to persist: recurring patterns, solved problems, project decisions, environment quirks
- Where: STATE.md for project-level decisions, CLAUDE.md for personal patterns, LESSONS.md for reusable insights
- When to persist: after solving a non-trivial problem, after making an architectural decision, after discovering a recurring pattern
- Description: "Pattern and error persistence across sessions. Defines what to persist, where to store it, and when to capture. Use when encountering recurring patterns, solving non-trivial problems, making architectural decisions, or discovering environment-specific behaviors."
