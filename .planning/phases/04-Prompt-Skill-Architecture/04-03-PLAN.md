The 4 agents replace 14 specialized agents:
- **executor** replaces: maxsim-executor
- **planner** replaces: maxsim-planner, maxsim-roadmapper, maxsim-plan-checker

Key principle: Agents are GENERIC. Specialization comes from:
1. Orchestrator spawn prompt (rich task-specific instructions)
2. Skills preloaded via frontmatter (shared protocols)
3. Skills auto-discovered on-demand (domain methodology)

CRITICAL RESEARCH FINDINGS:
- The `skills` field in agent frontmatter preloads FULL content at startup (not just metadata). Budget: 2-3 preloaded skills max per agent.
- Subagents CANNOT spawn other subagents. Orchestrator mediates all agent-to-agent communication.
- No custom MAXSIM-specific frontmatter extensions. Use only native Claude Code fields: name, description, tools, model, skills.
- Max 2 levels of @-references deep (command -> workflow -> inline content). Agent prompts must NOT @reference other files.

**Frontmatter (native Claude Code fields only):**
```yaml

name: <agent-name>
```

**Body structure for all agents (keep concise -- 60-120 lines total including frontmatter):**
1. One-line role statement
2. Input Validation section (gate: check required inputs, fail with structured error)
3. Core Protocol section (agent-specific execution steps)
4. Deviation/Error Handling section
5. Completion section (structured return using handoff-contract format)
6. Agent-specific gates (pre-action for executor, completion for verifier, etc.)

**Do NOT include in agent body:**
- agent_system_map (extracted to skill, auto-discovered on demand)
- Full handoff contract template (preloaded via skills field)
- Detailed verification methodology (preloaded via skills field or auto-discovered)
- @references to other files (max 2 levels is enforced; agent body is already level 2 since it's loaded by command->workflow)

**Agent specifications:**

**1. executor.md:**
- Role: Implements plans with atomic commits, deviation handling, and verified completion
- Tools: Read, Write, Edit, Bash, Grep, Glob
- Preloaded skills: handoff-contract, evidence-collection, commit-conventions
- Protocol: For each task: read spec -> implement -> run verify block -> produce evidence block -> commit -> next task
- Deviation rules: auto-fix bugs in touched files, cosmetic improvements if file already touched, scope creep logged as deferred, architectural changes STOP and return checkpoint
- Input validation gate: PLAN.md path must exist and be readable
- Pre-action gate: before each commit, verify task's done criteria with evidence
- Completion gate: before returning, verify all tasks attempted with evidence
- Description: "Implements plans with atomic commits, verified completion, and deviation handling. Use when executing PLAN.md tasks, making code changes, running build/test cycles, or implementing features from specifications."

**2. planner.md:**
- Role: Creates executable PLAN.md files with task breakdown, dependency analysis, and goal-backward verification
- Tools: Read, Bash, Grep, Glob
- Preloaded skills: handoff-contract, input-validation
- Protocol: Load context -> identify phase -> break into tasks -> build dependency graph -> assign waves -> group into plans -> derive must_haves -> write PLAN.md
- The planner does NOT use Write/Edit tools -- it writes PLAN.md files via Bash (this matches current behavior where planner uses Write tool for PLAN.md specifically)
- Actually, include Write tool -- planner needs it to create PLAN.md files
- Tools: Read, Write, Bash, Grep, Glob
- Input validation gate: ROADMAP.md and REQUIREMENTS.md must exist
- Completion gate: all PLAN.md files written with valid frontmatter
- Description: "Creates executable phase plans with task breakdown, dependency analysis, and goal-backward verification. Use when planning phases, creating PLAN.md files, breaking work into tasks, or performing gap closure planning."

**3. researcher.md:**
- Role: Investigates technical domains, evaluates sources, and produces structured findings
- Tools: Read, Bash, Grep, Glob, WebFetch
- Preloaded skills: handoff-contract, evidence-collection
- Protocol: Define questions -> identify sources -> research (prioritize official docs > codebase > community) -> evaluate confidence -> structure findings -> identify open questions
- Input validation gate: research topic/domain must be specified
- Completion gate: findings structured with confidence levels and sources cited
- Covers all research types: phase research, project research, codebase mapping, synthesis
- Orchestrator differentiates via spawn prompt (e.g., "Research the authentication domain" vs "Map the codebase structure")
- Description: "Investigates technical domains with structured source evaluation and confidence levels. Covers phase research, project research, codebase mapping, and synthesis. Use when researching libraries, APIs, architecture patterns, or any domain requiring external knowledge."

**4. verifier.md:**
- Role: Verifies work against specifications with fresh evidence, performs code review, spec review, and debugging
- Tools: Read, Bash, Grep, Glob
- Preloaded skills: verification-gates, evidence-collection, handoff-contract
- Protocol: Gather fresh evidence (NEVER trust prior claims) -> check each criterion -> produce evidence blocks -> pass/fail verdict
- This is the HARD GATE agent. Include anti-rationalization language directly in the agent body:
  - "Do NOT pass this gate by arguing it's 'close enough', 'minor issue', or 'will fix later'."
  - "Either evidence passes or it fails. No middle ground."
  - "Partial success is failure. 'Good enough' is not enough."
  - "REQUIRED: Cite specific tool call output as evidence. No tool output = no pass."
- Covers all verification types: phase verification, code review, spec review, debugging, integration checking, drift checking
- Orchestrator differentiates via spawn prompt (e.g., "Verify phase 4 success criteria" vs "Review this PR for code quality" vs "Debug this failing test")
- Input validation gate: verification criteria or review scope must be specified
- Completion gate: every criterion checked with evidence block, no criteria skipped
- Description: "Verifies work against specifications with fresh evidence. Covers phase verification, code review, spec review, debugging, and drift checking. Use when verifying phase completion, reviewing implementations, debugging failures, or checking spec compliance."
