<objective>
Remove the context monitor hook entirely (HOOK-04) and rewrite the statusline hook (HOOK-01). The context monitor is obsolete because Claude Code handles context natively. The statusline should display phase number and milestone progress sourced from a cached `gh api` call instead of the old context usage bar and task display.
</objective>

<execution_context>
This is Phase 6 of the MAXSIM v5.0 simplification. Hooks are standalone TypeScript files compiled by tsdown into CJS bundles in `dist/assets/hooks/`, then copied to `.claude/hooks/` at install time. The statusline receives JSON on stdin from Claude Code with model, workspace, context_window, and session_id. The context monitor reads a bridge file written by the statusline -- both the bridge file and the context monitor are being removed.

<action>
**1. Delete `maxsim-context-monitor.ts`** entirely. This file is the PostToolUse hook that reads bridge metrics and injects context warnings. It is fully replaced by Claude Code's native context handling.

**2. Rewrite `maxsim-statusline.ts`** with the following changes:

REMOVE:
- The entire context window display section (lines 26-63): the progress bar, the color coding, the bridge file write (`claude-ctx-{session}.json`). Claude Code handles context display natively.
- The "current task from todos" section (lines 66-88): reading agent todo files from `~/.claude/todos/`. Phase + milestone is sufficient.

ADD:
- A `ProgressCache` interface: `{ phase_number: string | null; milestone_title: string | null; milestone_pct: number; updated: number }`.
- Read phase/milestone data from a cache file at `path.join(dir, CLAUDE_DIR, 'cache', 'maxsim-progress.json')` where `dir` is `data.workspace?.project_dir || data.workspace?.current_dir || process.cwd()`. Use `project_dir` (not `current_dir`) for reliable project root detection per research findings.
- If cache exists and is fresh (age < 60 seconds), use cached values. If stale or missing, spawn a background `gh api` call that writes a new cache file, and use stale data in the meantime.
- The background refresh spawns a detached Node child process that runs `gh api repos/{owner}/{repo}/milestones --jq '.'`, parses the result to find the first open milestone, computes `closed_issues / (open_issues + closed_issues) * 100`, and writes the cache file. The owner/repo must be detected via `gh repo view --json nameWithOwner -q .nameWithOwner` inside the background process. Wrap the entire background spawn in try/catch -- if `gh` is not available, silently degrade.
- For the current phase number, use a second `gh` call in the same background process: `gh api repos/{owner}/{repo}/issues -f state=open -f labels=phase: -f per_page=1 -f sort=created -f direction=desc --jq '.[0].labels[] | select(.name | startswith("phase:")) | .name'` and parse the phase number from the label name (e.g., `phase:6` -> `"6"`).

CHANGE the update indicator path: read from `path.join(dir, CLAUDE_DIR, 'cache', 'maxsim-update-check.json')` instead of `path.join(homeDir, CLAUDE_DIR, 'cache', ...)` to support local-only install (ARCH-04).

KEEP the `readStdinJson` import from `./shared` and the standalone entry pattern.

Output format rules (from CONTEXT.md locked decisions):
- With MAXSIM project (`.planning/` exists): `[update] model | P{N} | v{M}: {pct}% | dirname`
- Without MAXSIM project: `[update] model | dirname`
- Update indicator: yellow up-arrow `\x1b[33m\u2B06\x1b[0m ` when update_available is true, otherwise empty string
- Phase/milestone segments use dim text (`\x1b[2m`), pipe separator ` \u2502 `

Update the `StatuslineInput` interface to include `project_dir?: string` in the workspace field.

**3. Update `index.ts`**: Remove the `processContextMonitor` export, `WARNING_THRESHOLD`, `CRITICAL_THRESHOLD`, `STALE_SECONDS`, `DEBOUNCE_CALLS` exports, and `ContextMonitorInput`/`ContextMonitorOutput` type exports. Keep `checkForUpdate`, `formatStatusline`, and their type exports.
</action>
