```bash
# 1. Check if .planning/ exists
PLANNING_EXISTS=$(test -d .planning && echo "true" || echo "false")

STATE=$(node .claude/maxsim/bin/maxsim-tools.cjs state load 2>/dev/null || echo "{}")
```

**Phase 2: Problem Detection** (check BEFORE suggesting action)

Check for and surface these problems (all blocking):
1. Uncommitted changes in .planning/ (state drift)
2. Blockers in STATE.md
3. Failed verification in latest phase
4. Stale GitHub Issues (local state diverged from GitHub)
5. Build failures (if `npm run build` or similar is configured)

If any problem found:
```
## Problem Detected

**Issue:** {description}
**Impact:** {why this matters}
**Resolution:** {what to do}

Resolve this before continuing. [Fix now / Investigate / Skip]
```

Block until resolved. Do NOT suggest next action until problems are cleared.

**Phase 3: Decision Tree** (explicit precedence order)

```
1. No .planning/ directory?
   -> "No MAXSIM project found."
   -> Act: /maxsim:init

2. Has blockers in STATE.md?
   -> "BLOCKED: {blocker text}"
   -> Act: surface blocker, suggest resolution

3. Has interrupted execution (from mcp_detect_interrupted)?
   -> "Phase {N} was interrupted: {X}/{Y} plans done."
   -> Act: /maxsim:execute N (resumes from incomplete)

4. Active phase has plan but not executed?
   -> "Phase {N} ({name}) is planned and ready."
   -> Act: /maxsim:execute N

5. Active phase needs planning?
   -> "Phase {N} ({name}) needs planning."
   -> Act: /maxsim:plan N

6. All phases complete?
   -> "All phases complete. Milestone ready."
   -> Act: /maxsim:progress

7. None of the above?
   -> Show interactive menu (3-4 contextual options + fallback)
```

Important: The go workflow should NOT use heavy MCP calls before showing something to the user. Show "Analyzing project..." immediately, then gather context.

```markdown
# MAXSIM Command Reference

## Table of Contents

1. [Getting Started](#getting-started)
2. [Commands](#commands)
3. [Plan Flow](#plan-flow)
4. [Execute Flow](#execute-flow)
5. [Quick Reference](#quick-reference)

## Getting Started

MAXSIM is a spec-driven development system for Claude Code. Work flows through milestones, phases, plans, and tasks.

**First time?** Run `/maxsim:init` to initialize your project.
**Returning?** Run `/maxsim:go` to auto-detect where you left off.

## Commands

### /maxsim:init
Initialize a new project or manage milestone lifecycle.
- New project: creates PROJECT.md, REQUIREMENTS.md, ROADMAP.md
- Existing project: scans codebase, creates planning docs
- Active project: offers milestone lifecycle options
Usage: `/maxsim:init [--auto]`

### /maxsim:plan [N]
Plan a phase through three stages: Discussion -> Research -> Planning.
- Discussion: gather implementation decisions via conversation
- Research: spawn researcher agent for domain analysis
- Planning: spawn planner agent to create executable PLAN.md files
- Auto-detects current stage and resumes from there
Usage: `/maxsim:plan 3` or `/maxsim:plan` (auto-detects next phase)

### /maxsim:execute [N]
Execute all plans in a phase with auto-verification.
- Runs plans in wave order (parallel within waves)
- Auto-verifies after execution
- Retries failed verification (max 2 retries)
Usage: `/maxsim:execute 3`

### /maxsim:go
Auto-detect project state and dispatch to the right command.
- Deep context analysis (GitHub Issues + local state + git)
- Surfaces problems before suggesting actions
- No arguments -- pure auto-detection
Usage: `/maxsim:go`

### /maxsim:progress
View project progress and milestone status.
- Phase completion overview
- Recent activity summary
- Offers milestone completion when all phases done
Usage: `/maxsim:progress`

### /maxsim:debug [description]
Systematic debugging with persistent state.
- Scientific method: hypothesize, test, verify
- Persistent debug sessions across context resets
- Spawns isolated debugger agent
Usage: `/maxsim:debug login form returns 500`

### /maxsim:quick [--full]
Ad-hoc tasks with MAXSIM guarantees.
- Atomic commits, state tracking
- --full flag enables plan-checking and verification
- Todo capture: "save for later" creates GitHub Issue
Usage: `/maxsim:quick` or `/maxsim:quick --full`

### /maxsim:settings
Configure MAXSIM workflow and model profile.
- Model profile (quality/balanced/budget)
- Research, plan checker, verifier toggles
- Branching strategy
Usage: `/maxsim:settings`

### /maxsim:help
Show this reference.

## Plan Flow

```
/maxsim:plan N
    |
    v
[Discussion] -- gather decisions via conversation
    |
    | Gate: "N decisions captured. Continue?"
    v
[Research] -- spawn researcher agent
    |
    | Gate: "Research complete. Continue?"
    v
[Planning] -- spawn planner agent
    |
    | Gate: "N plans in M waves. Ready?"
    v
  Done -- run /maxsim:execute N
```

Each stage produces artifacts:
- Discussion -> CONTEXT.md (locked decisions)
- Research -> RESEARCH.md (domain findings)
- Planning -> PLAN.md files (executable tasks)

## Execute Flow

```
/maxsim:execute N
    |
    v
[Execute] -- run plans in wave order
    |
    v
[Verify] -- spawn verifier agent
    |
    +---> PASS -> Phase complete!
    |
    +---> FAIL -> Retry (max 2)
              |
              v
          [Fix] -> [Re-verify]
              |
              +---> PASS -> Done
              +---> FAIL -> Report to user
```

## Quick Reference

| Want to... | Run |
|------------|-----|
| Start a new project | `/maxsim:init` |
| Plan next phase | `/maxsim:plan` |

| Debug an issue | `/maxsim:debug` |
| Quick ad-hoc task | `/maxsim:quick` |
| Change settings | `/maxsim:settings` |
| See this help | `/maxsim:help` |
```
