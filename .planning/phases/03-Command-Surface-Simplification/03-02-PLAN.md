<objective>
Create the unified `/maxsim:init` command that absorbs `/maxsim:new-project`, `/maxsim:init-existing`, and `/maxsim:new-milestone` into a single entry point. The init workflow is a THIN ROUTER that detects the project scenario and delegates to the existing sub-workflows (which already contain all the logic).

After this plan:
1. `templates/commands/maxsim/init.md` exists as the user-facing command
2. `templates/workflows/init.md` is a thin router (<15KB) that detects the scenario and delegates
3. The three existing workflows (`new-project.md`, `init-existing.md`, `new-milestone.md`) remain unchanged -- they contain the heavy logic
4. Scenario detection uses filesystem checks (.planning/ exists? ROADMAP.md exists? milestone complete?)
</objective>

<execution_context>
This is Phase 3 (Command Surface Simplification) of the MAXSIM v5.0 roadmap. This plan runs in parallel with Plan 01 (Wave 1) -- no file overlaps.

The `/maxsim:init` command unifies three separate commands into one. The key insight from RESEARCH.md is: do NOT try to merge the three workflows into one file. The `new-project.md` (46KB) and `init-existing.md` (47KB) workflows are massive. The init.md workflow should be a thin router that detects which scenario applies and delegates to the correct existing workflow.

Key constraints from CONTEXT.md:
- `/maxsim:init` on already-initialized project: AI detects current state -- completing milestone? Starting new one? Interactive flow
- Milestone lifecycle is interactive between user and AI
- Clean break, no v4.x migration

Scenario detection logic (from RESEARCH.md):
1. No `.planning/` directory -> NEW PROJECT (delegate to new-project.md)
2. `.planning/` exists, no ROADMAP.md -> EXISTING PROJECT (delegate to init-existing.md)
3. `.planning/` exists, ROADMAP.md exists, all phases complete -> MILESTONE COMPLETE (offer completion via new-milestone.md or start new milestone)
4. `.planning/` exists, ROADMAP.md exists, phases in progress -> ALREADY INITIALIZED (show status, offer options)

The existing workflows already reference their own @templates and @references. The init.md router just needs to load the right one.
</execution_context>

**Step 2: Create the thin router workflow** at `templates/workflows/init.md`.

This file MUST be under 15KB (target ~8-12KB). It is a ROUTER, not an implementation. Structure:

```markdown
# /maxsim:init Workflow

## 1. Detect Scenario

Check filesystem to determine which flow to run:

```bash
# Check if .planning/ exists
PLANNING_EXISTS=$(test -d .planning && echo "true" || echo "false")

# If exists, check for ROADMAP.md
ROADMAP_EXISTS=$(test -f .planning/ROADMAP.md && echo "true" || echo "false")
```

## 2. Route to Scenario

### Scenario A: New Project (no .planning/)
- Display: "No MAXSIM project found. Initializing new project..."
- Delegate to @./workflows/new-project.md
- After completion: suggest `/maxsim:plan 1`

### Scenario B: Existing Project (.planning/ exists, no ROADMAP.md)
- Display: "Existing project detected without roadmap. Initializing MAXSIM..."
- Delegate to @./workflows/init-existing.md
- After completion: suggest `/maxsim:plan 1`

### Scenario C: Active Milestone (.planning/ + ROADMAP.md, phases in progress)
- Load state: `node .claude/maxsim/bin/maxsim-tools.cjs state load`
- Display current status: milestone, phase progress, recent activity
- Offer interactive options:
  1. Continue working (suggest next action based on phase state)
  2. Start new milestone (delegate to @./workflows/new-milestone.md)
  3. View progress (`/maxsim:progress`)
  4. Exit

### Scenario D: Milestone Complete (all phases done)
- Display: "All phases complete! Milestone ready for completion."
- Offer options:
  1. Complete milestone and archive (delegate to new-milestone.md completion flow)
  2. Start new milestone (delegate to new-milestone.md creation flow)
  3. View progress (`/maxsim:progress`)
```

The workflow MUST:
- Load the @./references/ui-brand.md for consistent formatting
- Use the `@./templates/project.md` and `@./templates/requirements.md` templates (passed through to sub-workflows)
- Handle the `--auto` flag by passing it through to the sub-workflow
- For Scenarios C and D, use natural conversation for the interactive options (not AskUserQuestion)
- Reference but NOT inline the sub-workflows (they are 46KB+ each)
- Include a checkpoint-before-clear pattern for long-running flows

The workflow must NOT:
- Contain the full new-project logic (that stays in new-project.md)
- Contain the full init-existing logic (that stays in init-existing.md)
- Reference old command names (/maxsim:new-project, /maxsim:init-existing, /maxsim:new-milestone)
- Exceed 15KB

Note: The existing sub-workflows (new-project.md, init-existing.md, new-milestone.md) currently have their own "next steps" sections that reference old commands. Those will be cleaned up in Plan 07 (Wave 4 cross-reference update). For now, the init.md router just needs to delegate correctly.
