Frontmatter:
- `name: maxsim:execute`

`<execution_context>`: `@./workflows/execute.md`, `@./references/ui-brand.md`

`<context>`: `$ARGUMENTS` -- phase number (required). No `--gaps-only` flag (gap closure is handled automatically when verification fails and creates fix plans).

`<process>`: Execute the workflow end-to-end.

**Step 2: Create the execute workflow** at `templates/workflows/execute.md`.

Target size: 15-18KB. This is a state-machine orchestrator with the following stages:

**Stage 0: Initialize**
```bash
INIT=$(node .claude/maxsim/bin/maxsim-tools.cjs init execute-phase "$PHASE")
```
Extract: `phase_dir`, `phase_number`, `phase_name`, `plans` (array of plan files), `incomplete_plans`, `has_verification`, `has_uat`, `commit_docs`, `executor_model`.

**Stage 1: Detect State**
- All plans complete + verification passed -> ALREADY DONE (re-entry flow)
- All plans complete + no verification -> Start at VERIFY stage
- Some plans incomplete -> Start at EXECUTE stage (resume from incomplete)
- No plans exist -> ERROR ("No plans found. Run /maxsim:plan {N} first.")

**Re-entry flow** (when already executed + verified):
Show status summary. Offer options: (1) View results, (2) Re-execute from scratch, (3) View verification, (4) Done.

**Stage 2: Execute Plans**
Adapt from the existing `execute-phase.md` workflow:
1. Read all PLAN.md files, extract `wave` from frontmatter
2. Group plans by wave number
3. For each wave (sequential):
   a. Display wave header: "Wave {N}: {plan_count} plan(s)"
   b. For each plan in wave (parallel if multiple):
      - Spawn executor subagent via Task tool using the execute-plan.md sub-workflow
      - Pass the plan file path and execution context
   c. Collect results from all plans in wave
   d. If any plan failed: show failure details, ask user whether to continue or abort
4. After all waves complete: show execution summary gate
```
## Gate: Execution Complete
**Plans executed:** {N}/{total}
**Waves:** {M}
**Commits:** {list of commits}
Proceeding to verification...
```

**Stage 3: Auto-Verify**
Incorporate verification logic from `verify-work.md` and `verify-phase.md`:
1. Resolve verifier model: `node .claude/maxsim/bin/maxsim-tools.cjs resolve-model maxsim-verifier --raw`
2. Spawn `maxsim-verifier` agent via Task tool with phase context
3. Parse verifier return:
   - All truths verified -> PASS. Show verification gate:
     ```
     ## Gate: Verification Passed
     **Truths verified:** {N}/{N}
     **Evidence:** {summary}
     Phase {phase_number} complete!
     ```
   - Some truths failed -> FAIL. Show what failed and auto-retry.

**Stage 4: Retry Loop** (max 2 retries)
If verification failed:
1. Attempt counter: increment (max 2 retries = 3 total attempts)
2. If attempts remaining:
   - Display: "Verification failed. Retrying... (attempt {N}/3)"
   - Spawn planner in gap-closure mode to create fix plans
   - Execute fix plans (back to Stage 2 for gap plans only)
   - Re-verify (back to Stage 3)
3. If max attempts reached:
   - Display: "Verification failed after 3 attempts."
   - Show exactly what failed with evidence
   - Let user decide: manual fix, debug, or accept as-is

**Checkpoint pattern:**
Before recommending `/clear`, post checkpoint to GitHub Issue with current execution state (which plans completed, which wave, attempt number).

The workflow delegates per-plan execution to `execute-plan.md` (27KB) via @reference. The workflow must NOT inline the execute-plan logic.

Key differences from old execute-phase.md:
- Verification is built-in (was a separate command)
- Retry loop with gap closure (was manual)
- No `--gaps-only` flag (gap closure happens automatically in retry)
- Phase-level only (no plan-level execution)
- No references to old commands (/maxsim:execute-phase, /maxsim:verify-work)
